### 1.4 Current Limitations

- **No testing infrastructure**: The Makefile `test` target is a placeholder (`@echo "No tests configured yet"`)
- **No formal test framework**: No `tests/` directory, no pytest/unittest config, no test fixtures or expected outputs
- **No LLVM-specific validation**: No verification that the IR is correct, portable, or stable across compiler versions
- **The LLVM pipeline is separate**: Not integrated into the main Makefile targets; run independently via `python convert-to-llvm.py`

## 2. Proposed Directions

### 2.1 Tier 1 â€” Foundation (Highest Priority)

#### 2.1.1 Numerical Correctness: Cross-Compiler Validation

**Goal:** Confirm LLVM-compiled and GCC-compiled code produce identical numerical results (within FP tolerance).

**Approach:**
- Compile `experiments/test_scaling.c` against the `.ll` IR using clang:
  ```
  clang -o build/test_llvm experiments/test_scaling.c build/pfun_cma_engine.ll -lm
  ```
- Run both the existing GCC binary (`test_scaling`) and the new LLVM binary (`build/test_llvm`)
- Diff their output with appropriate floating-point tolerance
- Add as a Makefile target (`test-llvm`) and integrate into CI

**Concrete deliverables:**
- New Makefile target: `make test-llvm`
- Python or shell script to run both, capture stdout, and diff with tolerance
- Regression detection: any change to C source that alters numerical output will be caught

#### 2.1.2 Structural IR Validation (LLVM FileCheck)

**Goal:** Verify specific properties of the generated IR remain true as the C source evolves, detecting compiler behavior regressions cheaply.

**Approach:** Use LLVM's `FileCheck` utility to pattern-match the IR text for required properties.

**Key checks:**
- Auto-vectorization occurs in `calc_I_S`, `calc_I_E` (look for `<2 x double>` splat patterns)
- `llvm.fmuladd` intrinsics are present (FMA contraction)
- `llvm.assume` is present in `calc_G`
- All 12+ named functions exist in the module
- No unintended external function calls appear

**Concrete deliverables:**
- `.llcheck` file with FileCheck patterns
- Makefile target: `make check-ir`
- CI integration

#### 2.1.3 `.ll` â†’ Native Executable Pipeline

**Goal:** Formalize the end-to-end path from IR to runnable binary.

**Approach:**
```makefile
test-llvm: build/pfun_cma_engine.ll
	clang -o build/test_llvm experiments/test_scaling.c $< -lm
	build/test_llvm
```

This would provide the first `make test` target that actually does something useful.

### 2.2 Tier 2 â€” Exploration & Analysis

#### 2.2.1 LLVM `opt` Pass Exploration

**Goal:** Understand which LLVM optimization passes matter most for this numerical workload.

**Approach:** Run individual LLVM optimization passes on the IR and measure effects:
```bash
opt -passes=instcombine,bdce,gvn,slp-vectorizer -S build/pfun_cma_engine.ll -o build/opt.ll
```

**Metrics to track per pass:**
- Instruction count before/after
- Basic block count
- Whether SLP vectorizer adds value beyond clang's auto-vectorization
- Effect of `-O2` vs `-O3` vs `-Oz` on IR size and structure

#### 2.2.2 Cross-Target LLVM Backend Compilation

**Goal:** Demonstrate portability of the CMA model across architectures.

**Approach:** Use `llc` to re-target the IR to different architectures:
```bash
llc -mtriple=aarch64-linux-gnu    -o build/pfun_cma_engine-arm64.s build/pfun_cma_engine.ll
llc -mtriple=wasm32               -o build/pfun_cma_engine.wat    build/pfun_cma_engine.ll
llc -mtriple=riscv64-linux-gnu    -o build/pfun_cma_engine-rv64.s build/pfun_cma_engine.ll
```

**Value:**
- ARM64: Mobile deployment (iOS/Android health apps)
- WebAssembly: Browser-based simulation/demo
- RISC-V: Embedded/sensor applications

#### 2.2.3 IR-Level Static Analysis

**Goal:** Extract compute characteristics directly from the IR without running the model.

**Approach:** Use `opt -passes=print<cost-model>` and custom analysis scripts to quantify:
- Floating-point operation count per function (FLOP profile)
- Memory access patterns (sequential vs strided)
- Pointer aliasing assumptions (TBAA metadata usage)
- Vector lane utilization
- Estimated arithmetic intensity (FLOPs/byte)

### 2.3 Tier 3 â€” Advanced Applications

#### 2.3.1 ORC JIT Inline Compilation for Python

**Goal:** Use LLVM's ORC JIT to compile and execute the model at runtime, bypassing the system C compiler and enabling runtime specialization.

**Approach:** Write a CPython extension or use the Python `llvmlite` library to:
- Load the `.bc` bitcode
- Use ORC JIT to compile `run_cma_model` to native code
- Execute it from Python without an intermediate `.so` file
- Enable parameter-specific optimization (e.g., specialize for known `N_MEALS` values)

**Effort:** Medium-high. Requires understanding of LLVM's C API or llvmlite internals.

#### 2.3.2 GPU Offloading via LLVM

**Goal:** Target the computationally intensive loops (calc_G, calc_L) for GPU execution.

**Approach:** Two paths:
1. Add OpenMP `#pragma omp target` directives and use LLVM/Clang's offloading support
2. Use Polly (LLVM's polyhedral optimizer) to automatically detect and parallelize affine loops

**Feasibility:** The model has embarrassingly parallel loops over time points (1441 iterations) and meals (3 iterations). Kernel extraction is straightforward for the per-time-point signal computations.

**Effort:** High, but could yield significant speedup for batch simulation workloads.

#### 2.3.3 Bitcode Portability & LTO Testing

**Goal:** Verify the `.bc` bitcode loads correctly across LLVM versions and can be used for Link-Time Optimization.

**Approach:**
```bash
# Test bitcode round-tripping
llvm-dis build/pfun_cma_engine.bc -o /dev/null

# Test LTO with another LLVM-compiled module
clang -flto -O2 experiments/test_scaling.c build/pfun_cma_engine.bc -o build/test_lto
```

**Why it matters:** Bitcode stability across LLVM releases is a known pain point. Early detection of incompatibilities avoids surprise breakage when upgrading clang/LLVM.

#### 2.3.4 Optimization Regression Test Suite

**Goal:** Capture key IR metrics as "golden" values and alert on unexpected changes (e.g., after compiler upgrade).

**Metrics to track:**
| Metric | How to measure | Why it matters |
|--------|---------------|----------------|
| Total instruction count | `llvm-dis` + `wc -l` or `opt -passes=print<module>` | Overall code size |
| Vectorized loops | Count `!llvm.loop.isvectorized` metadata | Vectorization health |
| FMA intrinsics count | Count `llvm.fmuladd.*` calls | FMA contraction |
| External call count | Count `declare` non-intrinsic functions | Inlining effectiveness |
| Total basic blocks | `opt -passes=print<module>` | Indirect inlining metric |

## 5. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLVM tool version mismatch | IR incompatibility | Pin LLVM version in CI; document version requirements |
| Floating-point differences between GCC and clang | False positives in numerical tests | Use relative tolerance (e.g., `np.allclose(rtol=1e-12)`) |
| FileCheck patterns too brittle | Test maintenance burden | Test only stable IR properties (presence, not exact layout) |
| ORC JIT complexity | High effort, unclear ROI | Prototype with simplest case first (run_cma_model only) |
